> Última actualización previa: 4 May 2026 — Lote V.10.8 cerrado (singleton inter-contexto via globalThis + verificación empírica). Cero migración. Cero deps nuevas. Ataca el bug de **module isolation** detectado en V.10.7: aunque los jobs se encolaban OK (`iniciarCaptura · 7/7 encoladas`), el heartbeat del worker mostraba `waiting=0 active=0` justo después — confirmando que el route handler y el worker veían instancias DISTINTAS de la Queue (Next.js carga el módulo en contextos separados, cada uno con su propio module-level cache). **`cuotas-cola.ts`** ahora usa `globalThis as unknown as { __cuotasQueueInstance, __cuotasConnectionInstance, __cuotasModuleId }` como cache singleton inter-contexto. Patrón estándar de Next.js (mismo que el cliente Prisma). Helpers `getQueueInstance/setQueueInstance/getConnectionInstance/setConnectionInstance` reemplazan las variables module-level. **`cuotas-worker.ts`** recibe el mismo refactor: `globalForWorker.__cuotasWorkerInstance`, `__cuotasHeartbeatTimer`, `__cuotasWorkerModuleId` via globalThis. Helpers `getWorkerInstance/setWorkerInstance`. **Module IDs únicos** generados al primer load (`cuotas-cola-{ts}-{rand}`, `cuotas-worker-{ts}-{rand}`) — los logs los exponen en `moduleId` cuando se inicializa. Si ves dos `BullMQ — cola cuotas-captura inicializada` con el MISMO moduleId, el módulo se cargó dos veces pero la singleton funciona. Si ves IDs distintos, el globalThis no está compartiendo y hay otro bug. **Verificación empírica post-encolar** (`captura-cuotas.service.ts`): tras `iniciarCaptura`, lee `queue.getJobCounts(waiting, active, delayed, failed, completed)` y loggea `iniciarCaptura post-encolar · counts: waiting=N active=N ...`. Si los 7 jobs llegaron a Redis, `waiting >= 7` (asumiendo BullMQ no procesa al instante). Si `waiting=0` justo tras encolar, hay isolation incompatible — el escribe fue a un Redis distinto al que el handler lee. **Logs ricos** en `encolarJobCaptura` (debug level): por cada job encolado loggea `jobId`, `casa`, `partidoId`, `jobIdAsignado` (BullMQ devuelve), `connStatus` (ioredis status), `moduleId`. **Ruta de diagnóstico**: en el próximo smoke, después de activar Filtro 1, los logs deben mostrar (1) `iniciarCaptura · 7/7 encoladas`, (2) `iniciarCaptura post-encolar · counts: waiting=7` (o algún número >0), (3) `BullMQ active · job ... tomado por worker` × 7, (4) `procesando job ... vía playwright` × 7. Si la cadena se rompe entre (1) y (2), el problema es Redis/connection. Si entre (2) y (3), el worker no consume aunque haya jobs (posible queue distinta). Si entre (3) y (4), problema en el processor. Branch `feat/lote-v10-8-fix-module-isolation` → merge a `main` → push.

> Última actualización previa: 4 May 2026 — Lote V.10.7 cerrado (2 fixes de raíz + observabilidad rica). Cero migración. Cero deps nuevas. Ataca dos bugs detectados en el smoke V.10.6 + agrega telemetría rica a todo el flow Playwright para que los próximos fallos sean diagnosticables sin más iteraciones de código. **Fix 1: heartbeat** (`cuotas-worker.ts`): `worker.isPaused()` retorna boolean sincrono en algunas versiones de BullMQ pero mi código del V.10.4 hacía `.catch(() => null)` asumiendo Promise. Resultado: `heartbeat falló — a.isPaused(...).catch is not a function`. Fix con detección uniforme via `Promise.resolve()` + try/catch externo. **Fix 2: asset blocking** (`playwright-browser.ts`): removido `stylesheet` del `ASSET_TYPES_BLOQUEADOS`. El smoke V.10.6 mostró que las 7 casas devolvían 0 candidatos uniformemente — síntoma típico de CSS faltante que rompe layout/visibility de SPAs (componentes con `display: none` por default + clase CSS condicional, layout grid/flex, hidratación que espera CSS). Bloquear stylesheet ahorraba ~10-20% bandwidth pero rompía el render. Conservamos bloqueo de `image`/`media`/`font`/`imageset` que es el grueso del bandwidth sin afectar render. **Observabilidad rica en `playwright-scrape.ts`**: (1) log al `goto()` con `httpStatus` + `ms` + URL: `${casa} goto OK · {url} · status={N} · {N}ms`. (2) snapshot DOM post-hidratación con `titulo`, `h1`, `totalElementos`, `totalLinks`, `totalBotones`: permite distinguir "página real" vs "captcha/splash/geo-block" sin acceso al browser. (3) Cuando intento de búsqueda devuelve 0 candidatos, helper nuevo `diagnosticarDOM()` retorna `{totalClickables, conLocal, conVisita, conAmbos, muestrasConLocal[], muestrasConVisita[]}` — el log incluye 3 muestras de texto de elementos que tienen el local + 3 muestras con el visita. Permite ver si el partido está en el DOM con nombres distintos (alias necesario), si la página renderizó solo el local pero no el visita, o si el DOM está vacío. (4) Cuando extracción de cuotas falla, log con `totalBotones`, `elsConOdd`, `totalDecimales`, `titulo`, `h1` para entender por qué no se vieron las cuotas. **Heartbeat enriquecido** (`cuotas-worker.ts`): además de los counts BullMQ + paused/running, agrega `browserPlaywright: boolean` (browser warm vivo o no via `browserPlaywrightInicializado()`) y `rssMb: number` (memoria RSS del proceso). Permite diagnosticar OOM lurking, browser muerto, etc. Mensaje legible: `heartbeat · waiting=N active=N ... · paused=B running=B · browserPW=B · rss=NMB`. Trade-off: el fix 2 implica que cada captura va a tardar 1-2s extra (CSS download) y consumir +20-30MB RAM por page, aceptable. Branch `feat/lote-v10-7-fixes-y-observabilidad` → merge a `main` → push. Próximo smoke debería: (a) heartbeat ya no fallar, (b) si las casas siguen 0 candidatos, los logs nuevos van a decir EXACTAMENTE qué hay en el DOM (página completa o captcha o vacía o nombres distintos).

> Última actualización previa: 4 May 2026 — Lote V.10.5 cerrado (búsqueda iterativa Playwright + extracción por bloque). Cero migración. Cero deps nuevas. Reescritura de la heurística de scraping para emular el flow agentic que validó el admin con Claude Cowork: navegar la página, esperar hidratación SPA, scroll para forzar lazy-load, click en tabs si hace falta, validar que el click realmente navegó al detalle, extraer cuotas por bloque de mercado. **Reemplaza la heurística genérica del Lote V.9** que recorría el DOM completo una sola vez y fallaba consistentemente en producción (smoke V.10.4: Stake llegaba al detalle pero no extraía mercados, las otras 6 casas no encontraban el partido en el listado). **`buscarPartidoIterativo`** (`playwright-scrape.ts`): hasta 4 intentos con acción correctiva entre cada uno. Intento 1: heurística DOM mejorada con dos estrategias en cascada — primero busca elementos pequeños que contengan AMBOS equipos (caso fácil), si falla busca elementos con SOLO el local y verifica que un ancestor (hasta 5 niveles) contenga el visita (caso SPAs con spans separados). Intento 2: scroll 600px down → retry. Intento 3: click en tab "Próximos"/"Upcoming"/"Todos"/"Hoy"/"Mañana" → retry. Intento 4: scroll 1500px down → retry. Cada intento loggea cuántos candidatos encontró + cuál eligió para diagnóstico de producción. **Validación post-click**: `urlAntes !== page.url()` o `waitForURL` con timeout 10s. Si la URL no cambió tras click, asume modal in-page y espera 2.5s antes de intentar extraer. Si href está disponible y es URL distinta, prefiere navegación directa (más confiable que click sintético). **`extraerCuotasIterativo`**: espera con `waitForFunction` hasta 8s a que aparezca al menos un decimal `>1` en algún botón (señal de que las cuotas hidrataron). Después busca por bloque de mercado: encuentra el bloque (section/div) que contiene el header del mercado en su primer tercio + al menos un decimal — heuristic de "header + cuotas en mismo bloque". Para cada uno de los 4 mercados (1X2 / Doble Oportunidad / Más-Menos 2.5 / BTTS) busca su bloque por keywords ("Resultado"/"1X2"/"Match Result"/"Ganador" para 1X2, "Doble Oportunidad"/"Double Chance" para doble op, "Goles"/"Total"/"Over Under" para totales, "Ambos Anotan"/"BTTS"/"Both Teams to Score" para BTTS). Dentro del bloque busca cuotas por label exacto + extracción del decimal del propio elemento, descendant `.odd`/`.value`/`.price`/`<b>`/`<strong>`, sibling next, o parent. Si bloques no funcionan, fallback a heurística global del Lote V.9 (búsqueda por label en todo el DOM). **`intentarClickearTabComun`** helper nuevo que detecta tabs por texto normalizado ("próximos", "upcoming", "todos", "hoy", "mañana", "today", "tomorrow", "pre-match", "prepartido"); valida que el elemento sea pequeño (rect < 250×80px = típico de un tab/chip). **Tipos `PlaywrightPage` extendidos** (`playwright-browser.ts`): agregados `waitForURL` y `waitForFunction` al tipo estructural mínimo (sin importar de `playwright-chromium` para mantener tsc sin la dep instalada). **Timeouts ampliados**: `TIMEOUT_LISTADO_DEFAULT_MS` 25s→30s, `TIMEOUT_TOTAL_DEFAULT_MS` 60s→90s para acomodar los 4 intentos + scroll + tab-clicks que la heurística iterativa puede tomar. Decisiones documentadas: el umbral de score Jaro-Winkler para aceptar match se mantiene en `UMBRAL_FUZZY_DEFAULT` (0.88), el match falso positivo se descarta si min(scoreLocal, scoreVisita) < umbral × 0.7 = 0.616 (tolera variantes pero no aleatorio); la búsqueda de bloque de mercado requiere keyword en primer tercio del texto del bloque + al menos un decimal en el resto (filtra headers de menú irrelevantes); el fallback global solo aplica si la búsqueda por bloque no devolvió cuota — coexiste sin pisar. tsc + lint pasan limpios. Branch `feat/lote-v10-5-busqueda-iterativa-playwright` → merge a `main` → push.

## Reglas duras

1. **Prod-first, no local.** No correr `pnpm dev`/`next build`/migrar BD/levantar Postgres en local. Validación pre-push: solo `pnpm tsc --noEmit` + `pnpm lint`. Validación funcional la hace Gustavo en `hablaplay.com` post-deploy.
2. **Migraciones con `--create-only`.** Generar el SQL pero no aplicar local. Aplicación pasa por `prisma migrate deploy` en el `CMD` del Dockerfile al arrancar.
   - **Backup manual pre-deploy sólo si la migración compromete integridad de Postgres**: renombres de columna, cambios de tipo, conversiones JSONB↔relacional, FKs movidas, drops de tabla con data productiva. `CREATE TABLE` / `ALTER TABLE ADD COLUMN` / `UPDATE` de backfill puro **no** requieren backup.
3. **Branch por lote.** `feat/lote-<letra>-<slug>` (ej: `feat/lote-k-foundation-v32`). Migraciones y merges a `main` los aplica Claude Code directamente al cerrar el lote, sin gate de OK escrito de Gustavo. **Cierre incluye `git push origin main`** — Railway deploya desde `origin/main`, no desde local; merge sin push = deploy invisible.
4. **Commits Conventional.** `feat:`, `fix:`, `chore:`, `docs:`.
5. **Cero servicios externos nuevos** sin discutir antes. Las dependencias del v3.1 (OpenPay, WhatsApp Business, Anthropic, PageSpeed Insights) ya están aprobadas y son las únicas externas. Cualquier otra requiere discusión previa.
6. **TypeScript strict + Zod en entrada + Pino para logs** (no `console.log`).
7. **Cero hex hardcodeados en JSX.** Tokens Tailwind del Lote A (`brand-*`, `gold-*`, `premium-*`, `whatsapp-*`, `admin-*`).
8. **Fechas con timezone explícito.** Helpers en `lib/utils/datetime.ts`. Default `America/Lima`.
9. **`authedFetch` para `/api/v1/*`** desde el cliente. Centraliza `credentials: 'include'`.
10. **Modales con `createPortal(document.body)`** (`components/ui/Modal.tsx`).
11. **Operaciones admin one-shot** como endpoints `POST /api/v1/admin/*` con auth ADMIN o `Bearer CRON_SECRET`. Nunca en `startCommand`/`Dockerfile`.
12. **UX alineado al mockup v3.2.** Cualquier cambio de UX (copy, layout, componente, rebrand) respeta lo definido en `docs/habla-mockup-v3.2.html`. **El mockup v3.2 manda en cualquier ambigüedad y NO se modifica para acomodar la implementación** — al revés: la implementación se adapta al mockup.
